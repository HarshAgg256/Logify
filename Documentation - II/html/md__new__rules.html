Algorithm of the Code And Code Design

First off, we are taking no_of_lines and the proof as input from the user. Before even getting into the proof validity, It might be the case that the propositional formulas typed by the user are themselves not valid or possesses some incorrectness. So, we are applying a check for the same using formula_validity() function. This function takes note of equal number of opening and closing brackets so as to generate parenthesization errors. Moreover, it carries out a loop to hover over each character of the formula string and checks its neighbouring characters. Depending upon a given current character, its neighbouring characters can only be one of the few valid ones. If an invalid neighbouring character is encountered, false is returned. Else, if all the neighbouring characters of the current character are valid, next iteration is triggered via continue statement to hover over the next character. Encountering zero invalidities leads to return of true value. If any one of the lines returns false, the program terminates printing the line number on which incorrectness exists. As soon as the propositional formulas are proven valid, the control goes on to our checker() function wherein the main logic of our program sits. It contains a switch case which inspects the proof rule used on the current line number. When proof rule character matches with one of the cases, that rule is implemented on the propositional formulas written in line numbers specified after proof rule in the current. It returns the expected output and if it matches with the propositional formula of our current line, a true or false return value is generated accordingly. If a false value is being returned for any of the proof lines, We print "The Proof is invalid. Line number < > possesses some incorrectness". Otherwise, we print "The proof is valid".

Extension to Proof Rules

This logic can be easily extended to other proof rules as well just by adding new cases in the switch case functionality. Let's take an example of the new rule : IMPLICATION INTRODUCTION. It will be constructed under the already made case '>'. The nested switch case checking for the next character to be 'e' or 'i' can implement the introduction case too. Moreover, in the outer switch case, case 'A' denoting assumption must also be created. Using the already implemented proof rules, user can prove anything he/she wants (say q) starting from an assumption(say p). After the ending line of the assumption box, we can check if he has entered (p>q) or not depending upon which, his/her proof stands valid or invalid. Let's take another example of the new rule : DOUBLE NEGATION ELIMINATION. A new outer case '~' has to be created beneath which a nested switch case '~' has to be constructed. Again the control goes on to the propositional formula of line number specified in the current line after proof rule. In that formula, just consider its part after '~~'. If this substring matches with the propositional formula of our current line, his/her proof stands valid, otherwise it stands invalid.
